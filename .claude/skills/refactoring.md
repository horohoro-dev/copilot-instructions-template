# リファクタリング

## 黄金律

1. **振る舞いを変えない** — 「何をするか」ではなく「どうやるか」だけを変える
2. **小さなステップ** — 一度に1つの変更、変更ごとにテスト実行
3. **テスト必須** — テストなしのリファクタリングは「編集」であって「リファクタリング」ではない
4. **バージョン管理を活用** — 安全な状態ごとにコミット
5. **1つのことだけ** — リファクタリングと機能追加を混ぜない

## リファクタリングしてはいけない場面

- 動いていて今後変更されないコード
- テストのないクリティカルな本番コード（先にテストを書く）
- 厳しいデッドライン下
- 明確な目的がない場合

## 代表的なコードスメルと対処法

| コードスメル | 対処法 |
|------------|--------|
| 長い関数（50行超） | 関数抽出 |
| 重複コード | 共通ロジック抽出 |
| 大きなクラス | クラス分割 |
| 長いパラメータリスト | パラメータオブジェクト |
| マジックナンバー | 名前付き定数 |
| 深いネスト（5階層超） | ガード句・早期リターン |
| デッドコード | 削除 |

## 安全なプロセス

1. **準備**: テスト確認、コミット、ブランチ作成
2. **特定**: コードスメル発見、動作理解、計画立案
3. **実行**: 小さな変更 → テスト → コミット → 繰り返し
4. **検証**: 全テストパス、パフォーマンス影響なし
5. **クリーンアップ**: コメント・ドキュメント更新

## 代表的なリファクタリング操作

| 操作 | 説明 |
|------|------|
| メソッド抽出 | コード断片をメソッドに分離 |
| クラス抽出 | 振る舞いを新しいクラスに移動 |
| インライン化 | メソッドの中身を呼び出し元に展開 |
| 変数/メソッドの名前変更 | 明確性を向上 |
| パラメータオブジェクト導入 | 関連パラメータをグループ化 |
| 条件分岐のポリモーフィズム化 | switch/if をポリモーフィズムに置換 |
| ガード句導入 | ネストした条件分岐を早期リターンに置換 |

## チェックリスト

### コード品質
- [ ] 関数は小さい（50行以内）
- [ ] 各関数は1つのことだけを行う
- [ ] 重複コードがない
- [ ] 名前が説明的
- [ ] マジックナンバーがない
- [ ] デッドコードが削除されている

### 構造
- [ ] 関連コードがまとまっている
- [ ] モジュール境界が明確
- [ ] 依存関係が一方向
- [ ] 循環依存がない

### テスト
- [ ] リファクタリング対象にテストがある
- [ ] エッジケースがカバーされている
- [ ] すべてのテストが通る
