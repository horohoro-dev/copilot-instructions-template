# デバッグ

## 基本原則

- 推測ではなく証拠に基づいて行動する
- 一度に1つの変数だけを変える
- 修正前にバグを再現するテストを書く
- 修正後にリグレッションテストとして残す

## デバッグプロセス

### Step 1: 再現（Reproduce）

- バグを確実に再現できる手順を確立する
- 最小再現ケースを作成する
- 再現条件を明確にする（環境、データ、タイミング等）

### Step 2: 仮説（Hypothesize）

- 症状から原因の仮説を立てる
- 仮説を検証可能な形にする
- 複数の仮説がある場合は優先順位をつける
  - 最も可能性が高いものから検証
  - 最も検証しやすいものから検証

### Step 3: 検証（Verify）

- 仮説を1つずつ検証する
- テストコードで検証結果を記録する
- 仮説が誤っていた場合は新しい仮説を立てる

### Step 4: 修正（Fix）

- 根本原因を修正する（症状の対処ではない）
- 修正が他の部分に影響しないことを確認する
- リグレッションテストを追加する

### Step 5: 振り返り（Reflect）

- なぜこのバグが発生したのか
- どうすれば防げたか
- 類似のバグが他にないか確認する

## pytest デバッグ機能

```bash
pytest --pdb      # 失敗時にデバッガーを起動
pytest --lf       # 最後に失敗したテストだけ再実行
pytest -x         # 最初の失敗で停止
pytest -k "name"  # 特定のテストだけ実行
pytest -vv        # 詳細な出力
```

## 例外テスト

```python
import pytest

def test_無効な入力でValueError():
    with pytest.raises(ValueError, match="無効な値"):
        process_input(-1)
```

## やってはいけないこと

- print デバッグを本番コードに残す
- 推測だけで修正を試みる
- 複数の変更を同時に行う
- 再現手順を確立せずに修正する
- テストなしで修正を完了とする

## よくあるバグパターン

| パターン | 調査方法 |
|----------|----------|
| N+1 クエリ | Django Debug Toolbar / クエリログ |
| 競合状態 | ログのタイムスタンプ分析 |
| メモリリーク | memory_profiler |
| 型エラー | 型チェッカー（ty / mypy） |
| 境界値エラー | パラメータ化テスト |
