# インラインコメント規約

## 原則: What ではなく Why を書く

コードを読めば分かる処理内容の説明は不要。なぜその実装なのか、どんな制約・判断があるのかを書く。

```python
# Bad: 処理内容の説明（コードを見れば分かるコメントは不要）
df = pd.read_csv("data.csv")  # CSVを読み込む
result = price * 1.1  # 1.1を掛ける
user = None  # Noneを代入

# Good: 判断理由・背景の説明
result = price * 1.1  # 消費税10%を加算（2019年改正税率）
# Chrome Bug #1234 の回避策 — offsetHeight が hidden 要素で 0 を返す
height = element.scrollHeight
```

## コメントを書く前に

1. **変数名・関数名の改善**で意図を表現できないか検討する
2. **関数抽出**で処理の意図を名前にできないか検討する
3. それでも伝わらない Why がある場合にのみコメントを書く

```python
# Bad: コメントで補足が必要な命名
d = 7  # 有効期限の日数

# Good: 命名で意図を表現（コメント不要）
expiration_days = 7
```

## コメントが必要な場面

- **ワークアラウンド・バグ回避**: Issue 番号も記載する
- **ビジネスロジックの根拠**: 仕様書・要件への参照
- **非自明なパフォーマンス選択の理由**
- **一見不自然だが意図的なコード**
- **重要な副作用や制約の警告**
- **外部リソースへの参照**: RFC、仕様書、Stack Overflow 等

```python
# 挿入ソートを使用 — 入力は常に20件未満のため O(n²) でも
# クイックソートより定数係数が小さく高速（ベンチマーク: PR #456）
sort_small_list(items)

# このトリム処理は必須 — 上流APIが末尾にNULLバイトを含む文字列を返す
name = raw_name.strip("\x00")
```

## 禁止事項

- **コメントアウトしたコードを残さない** — Git で履歴管理する
- **セクション区切りのバナーコメント** — `######` 等の装飾は不要
- **変更履歴コメント** — Git log で管理する
- `# TODO:` / `# FIXME:` は許可するが、可能な限りマージ前に解消する
